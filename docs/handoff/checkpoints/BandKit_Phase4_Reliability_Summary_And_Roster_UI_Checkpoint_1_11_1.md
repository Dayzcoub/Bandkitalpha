## Scope of this slice

### Backend — party reliability summary (read side)

```text
server/src/modules/reliability/reliability.routes.js   (handlePartyReliabilitySummary)
server/src/index.js                                    (route wiring)
```

- `GET /api/v1/parties/:partyId/reliability-summary`.
- Deliberately NOT a rating number (Reputation Rules MVP). Returns structured counts:
  - `totals`: `{ positive, neutral, negative, disputed }`;
  - `by_type`: per-type counts with polarity and disputed sub-count.
- Disputed records are held OUT of the polarity totals and surfaced as `disputed` ("not final public negative reputation until reviewed").
- Visibility is layered: platform staff also see `moderation`-only records; everyone else sees `organizers` + `collaborators`; `hidden` is never aggregated.
- Authorization (verified-context rule, not arbitrary visitors): the viewer must be
  - platform staff, OR
  - the party's own owner (individual party's user, or an entity they manage), OR
  - a manager of an entity that has actually engaged this party;
  - otherwise `403`.

### Frontend — reliability on the engagements roster

```text
src/modules/RealEventEngagements.ts
src/locales/en/events.json
src/locales/ru/events.json
src/locales/bundles.ts                (regenerated by build)
```

- On `/events`, each engagement row gets a "Reliability (n)" toggle that lazily loads and expands an inline panel: existing records (type, reason, disputed marker, note) plus a record form (type, optional reason, note, "mark disputed").
- Records are written through the existing `POST /events/:eventId/engagements/:engagementId/reliability`; the list uses the matching `GET`.
- Built from existing shared UI classes only — no bespoke CSS, no inline styles (design-token rule). New i18n keys under `events.reliability.*` in en + ru (no hardcoded user strings).

## Verification

Backend (curl, authenticated session):

```text
GET summary — self/owner of party            -> 200, structured totals
GET summary — unrelated user                 -> 403 RELIABILITY_SUMMARY_FORBIDDEN
GET summary — no session                      -> 401 AUTH_REQUIRED
GET summary — unknown party                   -> 404 PARTY_NOT_FOUND
disputed record                               -> counted in totals.disputed, not negative
moderation-visibility record                  -> hidden from non-staff summary
```

Frontend (real browser via serve-dist proxy → backend, logged in as demo-manager):

```text
- /events engagements roster renders the reliability toggle per engagement;
- expanding loads the catalogue (12 types, 8 reasons) and existing records;
- recording a new event via the form appends it, bumps the toggle count, shows success;
- summary cross-check after UI record: positive:2, disputed:1, negative:0
  (moderation-visibility record correctly excluded from the non-staff self view);
- no console errors; clean mobile render.
```

## Deferred (next slices)

## Do not regress

- Keep the party summary structured (counts), never a single public rating, until anti-abuse (dispute + weighting) is designed.
- Keep disputed records out of polarity totals.
- Keep summary visibility layered; only platform staff see `moderation`-only records.
- Keep the roster UI built from shared classes / i18n keys — no inline styles, no hardcoded strings.
