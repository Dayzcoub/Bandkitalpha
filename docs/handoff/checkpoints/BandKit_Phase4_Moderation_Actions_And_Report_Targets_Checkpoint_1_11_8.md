### Backend — case actions

- Migration `0014`: `moderation_action_types` (no_action / warning / hide_content / restrict_user / suspend_user) + `moderation_actions` history table (`reason` NOT NULL by policy — the spec requires a reason for every sensitive action).
- `POST /api/v1/reports/:id/actions` (platform staff only). **Effects are real, not decorative:**
  - `hide_content` — sets the reported chat message `status='hidden'` (and unpins); message lists already exclude hidden.
  - `restrict_user` — sets target `users.status='restricted'`; `PermissionService` now rejects sanctioned statuses in `canWriteMessage` / `canCreateEntity` (restricted = read-yes, write-no).
  - `suspend_user` — sets `users.status='blocked'` (login and write paths already reject blocked).
  - `warning` / `no_action` — recorded in the history only.
- Target for user-level actions: explicit `target_user_id`, else the accused, else the author preserved in the evidence snapshot. **Staff and self cannot be sanctioned** through the case flow (staff management is a separate dual-approval concern). Every action writes `moderation.action.taken` to audit; effect + record are one transaction.
- `GET /reports/:id` now includes the case's action history.

### Frontend

- **Staff** ([/moderation] queue): each case gets a "Detail" toggle → preserved evidence, reporter details, action history (action / reason / target / actor), and a take-action form: required reason input + Warn / Hide content (chat messages only) / Restrict user / Suspend user.
- **Reporters**: the mock profile report link is now a real affordance — `RealReportTarget.ts` handles any `[data-report-target]` element with a floating reason picker (reuses the chat menu's global CSS) → `POST /reports`. Wired on: other users' profile pages (`user_profile`, reasons incl. impersonation / fake profile) and feed post cards (`post`, a ⚑ chip in the metric row). Object ids are polymorphic, so today's mock ids and tomorrow's real ids both work; guests get a no-op.

## Verification

Backend (curl; dedicated offender account with a message in the demo room):

```text
action without reason                        -> 400 ACTION_REASON_REQUIRED
non-staff action                              -> 403
hide_content                                  -> 200; message gone from the room list;
                                                 report's evidence snapshot still intact

                                                 GET messages -> 200 (read stays)
suspend_user                                  -> 200; users.status = 'blocked'

unknown action key                            -> 400
GET /reports/:id                              -> actions history included
```

## Deferred

- Un-restrict / un-suspend (restore) flow and temporary (auto-expiring) suspensions.

---

## Do not regress

- Every moderation action requires a reason and writes an audit event; effect and record stay in one transaction.
- Staff/self cannot be sanctioned from a case.
- `restricted` keeps read access but not write; `blocked` rejects login.
- Report affordances post to /reports with polymorphic object ids; guests no-op.

---

## Next recommended work

Restore actions (un-restrict / un-suspend) with the same reason+audit discipline, or move to the next Phase 4 domain (feed).
