# BandKit — Client IP Trust Boundary Checkpoint 1.21.1

## Status

Accepted checkpoint. Микро-срез, без миграции. Решение владельца 2026-07-16:
`clientIp()` — единственный доверенный источник IP во всём проекте.

Не часть Anti-Spam. Это граница доверия платформы, и чинится она отдельно, потому что
касается аудита, логов, форензики и abuse detection — всего, а не одного домена.

---

## Дыра

`clientIp()` брала **первый** элемент `x-forwarded-for`:

```js
const fwd = req.headers['x-forwarded-for'];
if (typeof fwd === 'string' && fwd.length) return fwd.split(',')[0].trim();
```

А nginx на стейджинге собирает заголовок так:

```nginx
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;   # "$http_x_forwarded_for, $remote_addr"
```

`$proxy_add_x_forwarded_for` **дописывает** настоящий адрес справа, сохраняя всё
присланное клиентом слева. Первый элемент поэтому целиком под контролем отправителя.

**Доказано на живом пути логина**, а не выведено из чтения кода. Один и тот же запрос,
одна и та же БД:

```text
X-Forwarded-For: 1.2.3.4, 203.0.113.9

старый код → sessions.ip = 1.2.3.4     (подделка)
новый код  → sessions.ip = 203.0.113.9 (настоящий, дописанный nginx)
```

## Почему это чинилось до всякого IP-лимита

Два повода, и оба не про анти-спам.

**`sessions.ip` пишется с миграции `0003`** и выглядит как форензика, не будучи ею.
Форжированное значение в таблице неотличимо от настоящего.

**Лимит по ключу, которым управляет атакующий, хуже отсутствующего.** Он не просто
обходится: подставив чужой адрес, можно выжечь чужую квоту, и жертва получит 429 за
чужие действия. Это, кстати, один из аргументов, почему IP-лимиты вообще вне MVP (D5).

## Правило: доверяем пиру, а не заголовку

- Сокет пришёл **не** от доверенного прокси → клиент говорит сам за себя, его заявления
  игнорируются целиком, берётся адрес пира.
- Сокет от доверенного прокси → берётся **последний** элемент: его дописал наш nginx, и
  он равен `$remote_addr`, который nginx наблюдал сам. Всё левее — чужие слова.

`TRUSTED_PROXY_IPS`, по умолчанию `127.0.0.1,::1`. Дефолт подобран так, чтобы правка не
требовала трогать env на VPS: node слушает только loopback, поэтому единственный
возможный пир в проде — свой же nginx. Пустое значение отключает доверие вовсе.

**Зашитое допущение: ровно один доверенный хоп.** Появится CDN или LB перед nginx —
последний элемент станет адресом nginx, и функцию придётся менять вместе с конфигом.
Длина цепочки — факт о развёртывании, а не о запросе; другого способа нет.

## Обойти нельзя механически

`scripts/check-ip-sources.mjs` в `npm run check`: ничто вне `clientIp()` не читает
`x-forwarded-for`, `x-real-ip`, `.remoteAddress` и `req.ip`. Правило уже один раз
держалось на внимательности и не удержалось. Гард проверен на реальном обходе, а не
принят за рабочий по зелёному ответу.

## Do not regress

- **Правило одного источника — механическое, не социальное.** Гард в `npm run check`.
- **Последний элемент, не первый.** Первый — это то, что прислал клиент.
- **Доверие определяется пиром**, а не наличием заголовка.
- **Один хоп.** Второй прокси — это правка кода вместе с конфигом nginx.

## Still open

- **Старые `sessions.ip` не чинятся ретроактивно.** Всё, записанное до 1.21.1, может быть
  подделкой и как форензика не годится. Чистка — удаление данных, требует явного решения
  владельца по конкретному действию.
- **Конфиг nginx — точка отказа вне репозитория.** Корректность опирается на то, что
  каждый `location` с `proxy_pass` на node ставит
  `proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for`. Сейчас такой блок ровно
  один (проверено). Второй proxy-путь без заголовка вернёт подделку.
- **Первый потребитель IP появится в Security Layer следующей фазы** (D5): ограничение
  анонимных путей — `/auth/login`, `/auth/register`. К тому дню функция готова.
