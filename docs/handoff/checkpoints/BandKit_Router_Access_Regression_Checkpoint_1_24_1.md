# BandKit — Router Access Regression Checkpoint 1.24.1

## Status

Accepted checkpoint. Без миграции. **Разбор регрессии, доехавшей до стейджинга.**

Это не срез с новой функциональностью. Это запись о том, как F6 закрыл четыре маршрута,
которые обязаны были остаться открытыми, и почему собственная проверка F6 этого не
увидела.

---

## Что сломалось

Четыре read-маршрута, анонимных **по дизайну** с 1.15.3, стали отвечать `401`:

```text
GET /entities                 каталог
GET /entities/:entityId       карточка сущности
GET /entities/:entityId/posts посты сущности
GET /posts/:postId/comments   комментарии
```

Все четыре резолвят актора, **терпят `null`** и скоупят выдачу сами:
`allowedVisibilities` без актора возвращает `['public']`; `canViewEntity` с `actor = null`
пропускает только публичное. Комментарий смоука говорит это прямым текстом: «каталог
отвечает гостям — публичные сущности для того и существуют, чтобы их находили, — но
members-only сущность в анонимной выдаче не появляется никогда».

## Почему я это сделал

`access` для 92 маршрутов выводился из вопроса «зовёт ли хендлер `resolveSessionUser`».
**Звать её и гейтить по ней — разные вещи.** Эти четыре спрашивают, кто ты, чтобы
показать тебе меньше, а не чтобы отказать.

Ошибка не в невнимательности к одному маршруту, а в неверном признаке: он давал верный
ответ для 88 маршрутов из 92, и поэтому выглядел рабочим.

## Почему проверка была зелёной

Вот главное, ради чего этот чекпоинт существует.

`check-route-contracts` сверяет объявление с поведением: `authed` → аноним обязан получить
401. Для `authed` это **замкнуто само на себя**: 401 производит тот самый диспетчер,
который проверяют, и он делает это ДО хендлера. Тест доказывает, что роутер исполняет
объявление, а не что объявление верное.

Иначе говоря: **проверка не могла поймать ровно тот класс ошибок, который F6 реально мог
внести** — пережатое объявление. Она ловит только «объявлен public, а гейт есть».

Чекпоинт 1.22.0 при этом утверждал: «Проваливаются оба направления». Неправда, написанная
уверенно, — и она исправлена там же.

## Чем чинится

Вторая часть проверки, статическая: для каждого `authed`-маршрута читается тело его
хендлера. Если он зовёт `resolveSessionUser` и нигде не отвечает `AUTH_REQUIRED` — он
обслуживает гостя, и это либо `public`-маршрут, либо хендлер без проверки. Оба случая
обязаны попасть человеку на глаза.

Предел записан в самом файле честно: хендлер, гейтящий через хелпер (`requireRoomAccess`,
`requireTarget`), этой проверке не виден — в его теле нет `resolveSessionUser`. Проверка
ловит конкретный класс, а не доказывает корректность всех 92 объявлений.

## Негативный контроль отработал дважды

Первый прогон новой проверки был зелёным — и это ничего не значило: `/\bhandle\w+/`
матчит слово `handler:` из строки таблицы раньше, чем имя функции, поэтому проверка молча
не находила ни одного хендлера. **Зелёный означал регулярку, а не правило.**

После правки на `/\bhandle[A-Z]\w*/` обе регрессии падают поимённо.

## Verification

```text
GET /entities аноним (стейджинг)     200 — каталог снова отвечает гостю
GET /taxonomy аноним                 200
GET /me/feed аноним                  401 — гейт на месте
GET /admin/users аноним              401
GET /events/<нет такого> аноним      404
негативный: /entities → authed       падает поимённо
негативный: /posts → authed          падает поимённо
Checks + Deploy                       success
```

## Do not regress

- **`resolveSessionUser` в хендлере ≠ гейт.** Признак «зовёт ли» неверен; смотреть надо,
  отвечает ли он `AUTH_REQUIRED`.
- **Проверка, чей предмет производит проверяемое поведение, ничего не проверяет.**
  Спрашивать: «что именно этот зелёный ответ исключает?»
- **Четыре маршрута отвечают гостю намеренно.** Закрыть их «для единообразия» — сломать
  каталог.

## Still open

- **Статическая часть не видит хендлеров с хелперами.** Сегодня среди них пережатых нет
  (проверено), но признак неполный.
- **Обратный класс — «объявлен public, а должен быть authed» — не ловится ничем.**
  Динамическая часть увидит 200 и согласится; статическая молчит про public. Это дыра, а
  не косметика: единственная защита здесь — object-level проверка внутри хендлера, то есть
  ровно то, на что F6 не покушался. Стоит своей проверки.
- **Регрессия доехала до стейджинга.** Гейт в Actions её не поймал и не мог: он гоняет
  `npm run check`, а знал об этом только смоук — который выполняется уже ПОСЛЕ деплоя и
  миграций. Проверки уровня «что API отвечает анониму» стоило бы иметь до VPS, а не после.
