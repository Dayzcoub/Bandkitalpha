### Backend

- `GET /api/v1/me/reliability` — records about the authenticated user's own party (`parties.user_id = actor.id`), filtered to `organizers` / `collaborators` visibility (the subject is not shown the moderation-only / hidden layer). Each row carries type, polarity, reason, note, event title, `disputed` and `dispute_state`.

### Frontend

- A "My reliability" card on `/profile/me` (own profile, logged in) lists the records with their event context and dispute state.
- A record with no dispute gets a **Dispute** button that reveals an inline form (optional reason + optional statement) posting to `POST /reliability-events/:id/dispute`. Opening marks the record disputed so it drops out of reputation totals until resolved.
- Localized via `profile.reliability.*` (en/ru); shared classes only, no new CSS.

## Verification

Real browser (serve-dist proxy → backend), logged in as demo-manager (whose own party had records):

```text
- /me/reliability returns 4 records (moderation-only record excluded);
- the "My reliability" card renders on /profile/me with event context + dispute states;
- only the record without a dispute shows a "Dispute" button;
- opening it (reason "Documented conflict" + statement) -> record shows
  "dispute: open", button disappears, "Dispute opened." message;
- no console errors; clean mobile render.
```

The dispute loop is now end-to-end: subject opens here → organizer resolves on the roster (1.11.4) → summary reflects the outcome (1.11.3 / 1.11.1).
