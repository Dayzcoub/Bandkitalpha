# BandKit — Chat Exit and Membership Rules v1.0

## Status

Accepted product logic.

This document fixes the rules for leaving, hiding, muting and archiving chats in BandKit. The key decision: **there must not be one universal “leave chat” action for every chat type**. Exit behavior depends on the chat context and the entity that grants access.

---

## Core principle

A chat can be independent, or it can be a communication layer attached to another entity:

- user-to-user relationship;
- free group conversation;
- band/project/workspace;
- orchestra/organization;
- event;
- safety/moderation case;
- admin/role-based room.

If the chat is attached to an entity, the user usually should not leave the chat separately while still being a member of that entity. Otherwise the product creates a broken state: the user remains responsible for a project/event but no longer sees operational decisions.

---

## Chat type rules

| Chat type | Generic “leave chat”? | Correct primary actions |
|---|---:|---|
| Direct chat | No | Hide/delete dialog locally, block user, report user |
| Free group chat | Yes | Leave chat, mute, archive |
| Project/band chat | No while still a project member | Mute, archive, hide from list, leave project/band |
| Orchestra/organization chat | No while still an organization member | Mute, archive, leave organization if permitted |
| Event chat | No while still an event participant | Mute, archive, decline/leave event |
| Safety/report chat | No | Close case, archive, report, escalate, block contact |
| Admin/role chat | Conditional | Leave only if role/ownership constraints allow |

---

## 1. Direct chats

Direct chats do not use a “leave chat” model.

Allowed actions:

- hide dialog;
- delete dialog locally;
- clear local history;
- block user;
- report user;
- mute notifications;
- archive dialog.

Rules:

- Deleting/hiding a direct chat is local to the current user unless backend policy later says otherwise.
- Blocking should stop new direct messages from that user unless the platform safety flow requires otherwise.
- Reporting should preserve message anchors/context for moderation.

Recommended labels:

- `Скрыть диалог`
- `Удалить у себя`
- `Заблокировать пользователя`
- `Пожаловаться`

Do not use:

- `Покинуть чат`

---

## 2. Free group chats

Free group chats are the only standard case where a generic “leave chat” action is valid.

Allowed actions:

- leave chat;
- mute notifications;
- archive chat;
- invite users, if permission allows;
- report chat/user, if needed.

Rules:

- Leaving removes the user from future messages and notifications.
- History visibility after leaving is a product setting and can be decided later, but MVP can keep previous history visible locally.
- The chat should receive a system message, for example:
  - `Alex Rhythm покинул чат.`
- Rejoining requires an invite unless the chat is public inside a workspace.
- If the user is the only owner/admin, leaving is blocked until ownership is transferred.

Recommended labels:

- `Покинуть чат`
- `Отключить уведомления`
- `Архивировать`

---

## 3. Project / band chats

A project/band chat is part of project membership. A user should not leave the chat separately while still being an active project member.

Allowed actions while still a member:

- mute notifications;
- archive chat;
- hide from the main chat list until new activity;
- leave project/band, if permitted.

Rules:

- If the user remains a project participant, they keep access to the project chat.
- Leaving the project/band removes access to project chats according to backend permissions.
- Critical project messages, pinned decisions, documents and event logistics should remain attached to the project history.
- If the user has a required role, leaving the project may require confirmation or reassignment.
- If the user is a project owner/admin/manager, they may need to transfer role before leaving.

Recommended labels:

- `Отключить уведомления`
- `Скрыть из списка`
- `Архивировать`
- `Покинуть проект`

Do not use while membership remains active:

- `Покинуть чат`

---

## 4. Orchestra / organization chats

Organization chats follow the same principle as project chats, but with stronger role constraints.

Allowed actions:

- mute notifications;
- archive chat;
- hide from main chat list;
- leave organization, if allowed by role and policy.

Rules:

- Users with organization roles should not be able to silently leave required operational rooms.
- Admins/owners cannot leave if they are the only responsible account.
- Role-based rooms should be generated by membership/permissions, not manual chat membership alone.

Recommended labels:

- `Отключить уведомления`
- `Архивировать`
- `Покинуть организацию`
- `Передать роль`, when required before leaving.

---

## 5. Event chats

An event chat is part of event participation. Users should not leave the chat while still listed as event participants.

Allowed actions:

- mute notifications;
- archive chat;
- hide until new activity;
- decline participation;
- leave/cancel participation, if event rules allow.

Rules:

- If the user is still a participant/performer/crew member, they keep access to the event chat.
- To stop being in the event chat, the user must decline/leave the event.
- Leaving close to the event may require a reason.
- Organizers/managers should be notified.
- The event chat should receive a system message when appropriate, for example:
  - `Mira Voice отказалась от участия в событии. Причина: болезнь.`
- If the event is near, cancellation may trigger reliability/rating logic later.

Recommended labels:

- `Отключить уведомления`
- `Архивировать`
- `Отказаться от участия`
- `Покинуть событие`

Do not use while event participation remains active:

- `Покинуть чат`

---

## 6. Safety / report / moderation chats

Safety chats should not have normal leave behavior.

Allowed actions depend on role:

For regular users:

- close dialog locally;
- archive;
- block contact;
- report;
- add more context.

For moderators/admins:

- close case;
- escalate;
- assign/reassign;
- archive;
- mark resolved;
- restrict or block user.

Rules:

- Message anchors and case context must remain available for moderation.
- A user should not be able to destroy safety evidence by “leaving” the chat.
- External-link and payment-off-platform safety context should remain preserved.

Recommended labels:

- `Закрыть обращение`
- `Архивировать`
- `Пожаловаться`
- `Передать модератору`
- `Заблокировать контакт`

Do not use:

- `Покинуть чат`

---

## 7. Admin / role-based chats

Admin chats depend on role membership.

Allowed actions:

- mute notifications;
- archive;
- leave role chat only if the role is removed or transferred;
- transfer role, if required.

Rules:

- Admin-only chats are generated by role permissions.
- A user should not leave a mandatory admin chat while keeping the admin role.
- The last admin/owner cannot leave or remove themselves without assigning another responsible user.
- Audit log should record role transfer and admin exit.

Recommended labels:

- `Отключить уведомления`
- `Передать роль`
- `Снять с себя роль`
- `Покинуть админский чат`, only when permissions allow.

## UX behavior

### Chat menu action priority

For entity-bound chats, the primary menu should show:

1. `Отключить уведомления`
2. `Архивировать`
3. Context action:
   - project: `Покинуть проект`
   - event: `Отказаться от участия`
   - organization: `Покинуть организацию`
   - safety: `Закрыть обращение` / `Пожаловаться`
4. Dangerous actions separated visually.

### Confirmation modals

Actions that affect membership require confirmation:

- leaving project;
- leaving organization;
- declining event participation;
- leaving free group chat;
- deleting direct dialog locally;
- blocking user;
- closing safety case.

The confirmation text must explain what access will be lost and whether history remains available.

### System messages

System messages should be created for shared membership changes:

- user joined chat;
- user left free group chat;
- user declined event;
- user left project;
- admin transferred ownership;
- user was removed by admin/moderator.

Direct local actions like hiding/archiving a dialog do not need system messages.

---

## Backend model implications

Future backend should separate:

- `chat_rooms` — chat identity and type;
- `chat_room_members` — membership inside free chats;
- `project_members` / `band_members` — project/band membership;
- `event_participants` — event access;
- `organization_members` — org access;
- `chat_room_permissions` or policy resolver — computed access rules;
- `chat_system_messages` or typed messages for membership/audit events;
- `notification_preferences` — mute/archive/hide state per user;
- `moderation_cases` — safety/report context.

Important: for entity-bound chats, `chat_room_members` should not be the only source of truth. Access must be computed from the parent entity membership and permissions.

---

## MVP implementation recommendation

For the current mock frontend stage:

- Add chat menu items later based on `chatContextMeta.kind`.
- For free group chat mock, show `Покинуть чат`.
- For project chat mock, show `Покинуть проект` instead of `Покинуть чат`.
- For event chat mock, show `Отказаться от участия` instead of `Покинуть чат`.
- For direct chat mock, show `Скрыть диалог`, `Удалить у себя`, `Заблокировать`, `Пожаловаться`.
- For safety chat mock, show `Пожаловаться`, `Заблокировать`, `Архивировать`, and moderator actions by role.

Do not implement backend persistence until chat permissions and membership schema are ready.

---

## Accepted product decision

BandKit should support leaving chats, but **not as one universal action**.

The accepted rule:

- free group chat → `Покинуть чат`;
- project/band chat → `Покинуть проект/группу` or mute/archive;
- event chat → `Отказаться от участия` or mute/archive;
- direct chat → hide/delete/block/report;
- safety chat → close/archive/report/escalate;
- admin/role chat → role transfer/removal first, then exit if allowed.
