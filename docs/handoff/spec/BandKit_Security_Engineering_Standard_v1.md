# BandKit — Security Engineering Standard v1

## Статус

Обязательный инженерный стандарт безопасности. Источник истины для того, **как
пишется код**, чтобы каждая строка учитывала защиту от атак, взломов и утечек
данных. Security-by-design, а не аудит постфактум.

Соотношение с другими документами:
- **Дополняет** `BandKit_Security_AntiFraud_v1_0.md` — тот про abuse/fraud
  (verification levels, link policy, репутация, модерация). Этот — про
  **code-level уязвимости** (injection, broken access control, auth, crypto,
  SSRF, утечки данных).
- **Реализует** платформенные предохранители из ТЗ §20 на уровне правил кода.
- **Гейт `/security-review`** остаётся runtime-проверкой per-branch по диффу.
  Этот документ — проактивная сторона: код изначально пишется так, чтобы review
  не находил ничего.

## 0. Базовые принципы

1. **Сервер — единственный источник истины для прав.** Frontend-роли
   (`localStorage['bandkit.role']`) — только UX-хинт, НИКОГДА не проверка доступа.
2. **Secure by default.** Запрещено по умолчанию, разрешено явно. Новый эндпоинт
   без авторизации = закрыт, а не открыт.
3. **Deny, потом allow.** Доступ к объекту доказывается (membership + permission),
   а не предполагается.
4. **Минимум данных.** Отдаём/логируем/храним минимально необходимое.
5. **Каждый вертикальный срез проходит security-гейт §16 до чекпоинта.**

## 1. Аутентификация (AuthN)

- Пароли — только **argon2id** (или bcrypt cost≥12). Никогда не хранить/логировать plaintext.
- Сессии — серверные, cookie: `HttpOnly`, `Secure`, `SameSite=Lax/Strict`. Не JWT в localStorage.
- Ротация session id при логине и при повышении привилегий (step-up).
- **2FA (TOTP)** обязательна для владельцев workspace, админов, платных
  управленческих, super_admin, критичных действий (ТЗ §7).
- **Recovery codes** — показ один раз, хранение только в хеше, статусы used/revoked,
  пересоздание через step-up.
- Magic link / OAuth (Google/Apple) — токены одноразовые, короткоживущие,
  привязаны к устройству/сессии.
- Защита от перебора: rate-limit + прогрессивная задержка + captcha/risk challenge
  на подозрительный вход. Ответы login не раскрывают, «нет юзера» vs «неверный пароль».

## 2. Авторизация и изоляция арендаторов (AuthZ) — приоритет №1

Для workspace-SaaS это главный класс риска (broken access control / IDOR).

- **Каждый запрос к ресурсу проверяет object-level доступ** через серверный
  `PermissionService`: `entity_membership.role` × `platform_role` → capability.
- **Каждый запрос к данным workspace scoped по membership.** Запрещены запросы,
  которые не фильтруют по принадлежности актора. `WHERE entity_id = $1` без
  проверки, что актор — член этой entity, = дыра.
- **IDOR на `:id-or-slug`** — публичный идентификатор не даёт доступа. Проверка
  прав отдельно от резолва объекта.
- **Не мешать контексты** (ТЗ / handoff): `platform_role` ≠ `entity_membership.role`
  ≠ `support_case_assignment` ≠ `moderation_case_assignment`. platform_admin не
  становится admin каждой entity.
- **DTO-разделение public / private / admin** (ТЗ §21). Public API не отдаёт
  private/admin поля. Никогда не сериализовать доменный объект целиком в ответ.
- Опасные действия (ban, ownership transfer, trust reset, refund, mass delete,
  super_admin grant) — **two-person approval + reason + audit** (ТЗ §15).
- **Break-glass** доступ — временный, с причиной, логируется, review. Support
  view-as-user — только read-only.

## 3. Инъекции (SQL / command / template)

- **Только параметризованные запросы** (`pg` placeholders `$1,$2`). Запрещена
  конкатенация пользовательского ввода в SQL. Никогда не подставлять идентификаторы
  таблиц/колонок из ввода.
- Никаких `eval`, `Function()`, динамического `require` от ввода.
- Deploy/CI-скрипты (`scripts/*.sh`): не интерполировать неэкранированный ввод в
  shell; кавычить переменные; не логировать секреты.
- ORM/query-builder (если появится) — только с bound-параметрами.

## 4. Валидация ввода

- **Каждый эндпоинт валидирует вход схемой** (ТЗ §18: все формы через схемы) —
  тип, длина, формат, whitelist допустимых значений. Reject, не sanitize молча.
- Валидация на сервере — обязательна; на клиенте — только UX.
- Нормализация email/phone до проверок и хранения (ТЗ §20).
- Ограничения размера тела запроса, длины строк, размера массивов.

## 5. Вывод и XSS (frontend)

⚠️ Текущий frontend собирает HTML через template-строки (`badge()`, `listRow()`,
`kpi()` пишут innerHTML) и вручную зовёт `escapeHtml()`. Один пропущенный
интерполят пользовательских данных = stored XSS.

- **Любые пользовательские данные в HTML — только через единый safe-render хелпер**
  (экранирование по умолчанию). Не полагаться на то, что каждый вызов вручную
  вспомнит `escapeHtml`.
- Предпочитать `textContent` / создание нод вместо `innerHTML`, где возможно.
- Внедрить **Content-Security-Policy** (без `unsafe-inline`), `X-Content-Type-Options: nosniff`,
  `Referrer-Policy`, `X-Frame-Options`/`frame-ancestors`.
- URL из ввода — валидировать схему (`http/https` only, без `javascript:`), не
  вставлять в `href`/`src` без проверки.

## 6. Файлы и медиа

- **Whitelist MIME + магические байты**, не только расширение. Исполняемые/опасные
  типы запрещены (ТЗ §14).
- Лимиты размера; нормализация имени файла; запрет path traversal (`../`).
- Приватные файлы — только через **signed URL / proxy с permission check**. Не
  раздавать из webroot.
- Загруженные файлы — на отдельном origin / с `Content-Disposition: attachment`,
  чтобы не исполнялись как страница.
- Upload lifecycle (initiated→scanning→ready→rejected), antivirus/scan — шов на будущее.
- Orphaned uploads чистятся background job.

## 7. SSRF и внешние ссылки

- Любой серверный fetch по URL из ввода (OG-preview, link guard, аватар по URL) —
  **deny внутренних диапазонов** (127.0.0.0/8, 10/8, 172.16/12, 192.168/16,
  169.254.169.254, `::1`), запрет редиректов на них, таймауты, лимит размера.
- Link Guard (ТЗ §9) применяется ко всем полям с пользовательским текстом.

## 8. Секреты и конфигурация

- Секреты только в `server/.env` на VPS; **никогда в git**, коде, логах, ответах API.
- `.gitignore` покрывает `.env`, `.env.*` (уже есть). Проверять на утечку перед коммитом.
- **Логирование с redaction**: пароли, токены, session id, 2FA-коды, PII —
  маскируются (ТЗ §20).
- Ротация секретов; GitHub Actions/SSH-ключи — как secrets, не в workflow-файле.
- Разделение окружений local / staging / production.

## 9. Приватность и утечки данных (PII)

- **Data minimization + classification** (ТЗ §20). Отдавать только нужные поля.
- Public API / публичные страницы — `noindex` для приватного, anti-scraping лимиты,
  privacy-aware search (ТЗ §21).
- Защита обхода контактов (email/phone/QR в тексте и картинках) — ТЗ §6.
- Финансовая приватность; NDA/конфиденциальные проекты — отдельный scope.
- Encryption at rest для чувствительных данных.
- **Account deletion / export** не должны ломать историю entity (ТЗ Account Rules).

## 10. Криптография и E2EE-чаты

- Не изобретать крипто. Стандартные библиотеки, проверенные алгоритмы.
- **Личные чаты E2EE by default** (ТЗ §8): сервер хранит **только зашифрованный
  payload**, не читает plaintext. Жалоба — через добровольную передачу
  evidence-фрагмента пользователем.
- Системные события — отдельно от приватного user-текста.
- Ключи, соли, IV — генерируются криптостойко, не переиспользуются.

## 11. Rate limiting и DoS

- Лимиты по **user / IP / device / workspace / trust level** (ТЗ §20).
- **Пагинация с обязательным cap** на всех списках (нет unbounded query).
- Дорогие операции (экспорт, PDF, поиск) — отдельные лимиты/очереди.
- SMS/email — anti-pumping, deliverability tracking.

## 12. Транспорт и CSRF

- **Preview-VPS (`http://141.98.87.9`) — личный сервер под визуальные/функциональные
  проверки на mock-данных.** Пока там нет реального логина/паролей/PII, plain HTTP
  допустим — это не прод. **HTTPS/TLS становится обязательным до Фазы 1 (auth):**
  как только появляется реальная форма логина (включая вход самого super_admin) или
  любой реальный PII, пароль/сессия по HTTP уходят в открытом виде. Настройка
  дешёвая (Caddy / Let's Encrypt авто-TLS).
- HSTS после включения TLS.
- Cookie-based auth → **CSRF-защита**: `SameSite` + токен на state-changing запросы.

## 13. Аудит и логирование

- **Append-only / immutable** audit для sensitive/moderation/payment/terms (ТЗ §16).
- Логировать: изменение прав, админ-действия, просмотр админом sensitive-данных,
  security events, billing events.
- Запись audit — часть транзакции действия, не «best effort».
- Логи не содержат секретов/PII (см. §8).

## 14. Целостность денежных и фоновых операций

- Money-операции — **идемпотентны** (idempotency key), **outbox pattern**, dead
  letter queue (ТЗ §20). Никаких двойных списаний при ретрае.
- Feature-доступ — через **feature limits**, не `plan === 'pro'` (ТЗ §4).

## 15. Зависимости и цепочка поставки

- Минимум зависимостей (сейчас 0 runtime deps на frontend — держать так).
- Backend `pg` и любые новые deps — pin версий, аудит на известные CVE перед
  добавлением. Новая зависимость требует обоснования.
- Не тянуть deps ради того, что уже даёт stdlib/платформа.

## 16. Security Definition of Done (гейт каждого среза)

Вертикальный срез НЕ уходит в чекпоинт, пока не выполнено:

1. Все новые эндпоинты требуют auth (кроме явно публичных) и проходят через
   `PermissionService` (object-level).
2. Все запросы к данным workspace scoped по membership актора.
3. Весь пользовательский ввод валидируется схемой на сервере.
4. Весь вывод пользовательских данных экранируется (safe-render).
5. SQL — только параметризованный.
6. Ответы не содержат private/admin полей (DTO-разделение).
7. Секретов нет в коде/логах/ответах.
8. Sensitive-действия пишут audit.
9. Списки — с пагинацией и cap.
10. Ошибки не раскрывают внутренности (stack/SQL/пути) клиенту.
11. Изменения схемы — только миграцией.
12. `/security-review` по ветке — без находок high/critical.

## 17. Что исправить в фундаменте до появления auth

Проактивные долги, которые дешевле закрыть сейчас:

- **HTTPS/TLS на preview-VPS** — не сейчас (mock-only preview ок по HTTP), а до
  Фазы 1 / первого реального логина (§12).
- **Единый safe-render хелпер + CSP** на frontend вместо ручного `escapeHtml`
  по всем модулям (§5).
- **Серверный `PermissionService`** как источник истины до подключения любых
  write-флоу (§2) — совпадает с Фазой 2 плана разработки.
- Подтвердить, что `localStorage`-роли нигде не влияют на серверные решения (§0).

## 18. Не-цели этой фазы

Пентест, формальная threat-model на каждую фичу, антивирус-сканирование загрузок,
WAF — позже. Сейчас: правила кода + DoD-гейт, чтобы долг не накапливался.
