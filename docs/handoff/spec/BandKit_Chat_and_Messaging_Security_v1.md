# BandKit — Chat and Messaging Security v1.0

Статус: **обязательная продуктовая и архитектурная спецификация**.

Этот документ заменяет требования `BandKit_TZ_v1_2.md` §8 «Чаты и E2EE» и уточняет §9 «Link Guard» в части сообщений. При конфликте с более ранним ТЗ применяется этот документ.

## 1. Базовая модель переписки

В BandKit существуют только два независимых класса переписки:

1. личный диалог между двумя пользователями;
2. отдельный групповой чат сущности.

Личные диалоги и чаты сущностей не объединяются, не наследуют сообщения друг друга и не образуют промежуточных типов.

В системе нет `entity direct message`, «лички внутри группы», «лички внутри мероприятия» или отдельного диалога с одним человеком для каждого контекста.

## 2. Личный диалог user ↔ user

Личный диалог принадлежит двум пользователям платформы и не принадлежит группе, проекту, мероприятию, студии, организации или другой сущности.

Правила:

- для одной неупорядоченной пары пользователей существует не более одного канонического личного диалога;
- нажатие «Написать» на карточке пользователя из любого контекста всегда открывает этот канонический личный диалог;
- открытие пользователя из группы, события, каталога, комментария или другого экрана не создаёт отдельную переписку;
- сущность, её владелец, администраторы и другие участники не получают доступа к личной переписке пользователей;
- членство пользователей в одной сущности не меняет владельца, область доступа или жизненный цикл личного диалога;
- прекращение членства в сущности не удаляет и не переносит личный диалог.

Минимальные инварианты модели:

```text
conversation_scope = personal
entity_id = null
participant_count = 2
unique unordered participant pair
```

## 3. Чат сущности

Каждая поддерживаемая сущность может иметь собственный отдельный групповой чат.

Примеры:

- чат группы;
- чат проекта;
- чат мероприятия;
- чат студии;
- чат организации.

Правила:

- чат принадлежит ровно одной сущности;
- чат группы и чат мероприятия являются разными переписками даже при полностью совпадающем составе участников;
- сообщения не копируются и не синхронизируются между чатами разных сущностей;
- сообщения чата сущности не попадают в личные диалоги;
- общий чат сущности открывается отдельным действием со страницы сущности;
- нажатие «Написать» на участнике сущности открывает личный диалог user ↔ user, а не приватную ветку сущности;
- доступ определяется сервером по активному членству, роли и permission policy сущности;
- удалённый или утративший доступ участник больше не может читать новые данные чата;
- ранее отправленные сообщения участника могут сохраняться в истории сущности по её retention policy;
- системные timeline-события хранятся отдельно от пользовательских сообщений.

Минимальные инварианты модели:

```text
conversation_scope = entity
entity_type is required
entity_id is required
participants are derived from entity access policy
```

## 4. Состав чата сущности

Для группы в чат автоматически входят активные участники, которым роль разрешает доступ к групповому чату.

Для мероприятия доступ к рабочему чату не равен публичному интересу к событию. Зритель, подписчик или пользователь со статусом «пойду» не получает доступ автоматически.

В чат мероприятия могут входить только пользователи, допущенные его access policy, например:

- организаторы;
- подтверждённые участники;
- представители участвующих сущностей;
- техническая команда;
- приглашённые специалисты;
- иные явно добавленные доверенные участники.

Доступ всегда проверяется сервером; frontend-состояние не является источником истины.

## 5. Текущая модель шифрования

На текущем этапе оба класса чатов работают в `server_managed` режиме:

- TLS при передаче;
- шифрование хранилища и резервных копий;
- минимальный сервисный доступ;
- строгая object-level authorization;
- аудит административных и модерационных действий;
- отсутствие штатного чтения переписки через обычную админку вне утверждённого moderation flow.

Это **не E2EE**. Интерфейс, документация и маркетинговые тексты не должны называть текущие чаты end-to-end encrypted.

## 6. Серверная обработка сообщений

Поскольку текущий режим является `server_managed`, сервер имеет право обрабатывать открытый текст строго для заявленных функций:

- Link Guard;
- модерация и anti-abuse;
- создание evidence-снапшота при жалобе;
- серверный поиск, если он разрешён продуктовой политикой;
- уведомления и техническая доставка;
- выполнение legal/security obligations в пределах утверждённых процедур.

Доступ к открытому тексту не означает свободный административный просмотр. Любой staff-доступ должен быть минимальным, ролевым, обоснованным и аудируемым.

## 7. Link Guard

В текущем `server_managed` режиме Link Guard применяется сервером и к личным диалогам, и к чатам сущностей.

Правила:

- проверка выполняется до принятия сообщения либо в рамках явно определённого quarantine flow;
- blocked-сообщение не должно считаться доставленным;
- причина блокировки отображается пользователю без раскрытия внутренних detection rules;
- решение, risk type, actor, conversation scope и повторяемость события логируются;
- whitelist и false-positive flow управляются через утверждённую административную процедуру;
- Link Guard не должен предоставлять администраторам произвольный интерфейс чтения переписки.

## 8. Жалобы, evidence и модерация

В текущем `server_managed` режиме жалоба на сообщение может создавать неизменяемый evidence-снапшот сообщения и минимально необходимого контекста.

Evidence должен содержать только данные, необходимые для рассмотрения жалобы, например:

- идентификатор сообщения;
- автора;
- conversation scope;
- entity reference для чата сущности;
- текст или payload сообщения на момент жалобы;
- вложения или ссылки, связанные с жалобой;
- timestamps;
- ограниченный контекст соседних сообщений, когда он действительно нужен;
- технические moderation metadata.

Evidence:

- сохраняется на момент жалобы и не меняется вслед за редактированием или удалением исходного сообщения;
- доступен только разрешённым moderation roles;
- каждое чтение и действие аудируется;
- хранится по отдельной retention policy;
- не превращается в общий инструмент просмотра чатов.

## 9. Будущий E2EE

Полноценное E2EE не входит в текущий chat slice и оформляется отдельным архитектурным epic.

Допустимая будущая область E2EE — только личные диалоги user ↔ user. Чаты сущностей остаются `server_managed`, поскольку требуют управляемого состава, истории, Link Guard, moderation evidence, server-side permissions и lifecycle сущности.

До реализации E2EE модель данных может предусматривать точки расширения:

```text
encryption_mode: server_managed | e2ee
ciphertext
payload_version
key_version
sender_device_id
```

Наличие полей или enum-значения не означает, что E2EE реализовано.

Отдельный E2EE epic должен до включения режима спроектировать и проверить:

- генерацию ключей на клиенте;
- multi-device key distribution;
- добавление, подтверждение и отзыв устройств;
- key rotation;
- encrypted attachments;
- backup и recovery;
- честный UX потери истории при невозможности восстановления ключей;
- миграцию существующих личных диалогов;
- client-side search;
- client-side или privacy-preserving link protection;
- добровольную передачу evidence-фрагмента при жалобе;
- подтверждение происхождения переданного evidence;
- push-уведомления без утечки содержимого;
- threat model и отдельный security review.

До завершения этого epic серверная модерация, Link Guard и evidence не должны искусственно отключаться в личных диалогах.

## 10. UX-контракт

Интерфейс должен ясно различать:

- личные диалоги;
- чаты групп;
- чаты проектов;
- чаты мероприятий;
- чаты других сущностей.

Допускается единый список входящих с фильтрами, но это только presentation layer. На уровне данных и прав переписки остаются независимыми.

Кнопка «Написать» у пользователя всегда означает личный диалог.

Кнопка открытия чата сущности располагается в контексте соответствующей сущности и не должна выглядеть как личная переписка с отдельным участником.

## 11. Security Definition of Done для chat slices

Каждый chat slice обязан проверить:

- невозможность создать второй личный диалог для той же пары пользователей;
- невозможность привязать личный диалог к entity;
- невозможность создать entity-чат без entity reference;
- отсутствие доступа к entity-чату после утраты membership/permission;
- отсутствие доступа сущности или её администраторов к личным диалогам участников;
- server-side authorization для каждого read/write action;
- Link Guard на всех предусмотренных write paths;
- неизменяемость evidence-снапшота;
- audit trail moderation-доступа и действий;
- защиту от IDOR, cross-tenant leakage, XSS и unsafe attachment handling;
- отсутствие заявления E2EE в UI до фактической реализации полного протокола.
